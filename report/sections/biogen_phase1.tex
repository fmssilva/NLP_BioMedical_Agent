\section{BioGen NL Agent}
\label{sec:biogen}

\subsection{Data Parsing, Indexing and Search (Phase~1)}
\label{sec:phase1}

% ── 2a.1  CORPUS ──────────────────────────────────────────────────────────────
\subsubsection{Corpus}
The collection comprises 4{,}194 PubMed abstracts with a mean length of
150~words (median 142, range 12--964).  The high variance
($\sigma \approx 100$~words) across documents is relevant for retrieval
design: it motivates explicit length normalisation in BM25 (parameter~$b$)
and Dirichlet smoothing (parameter~$\mu$).

% ── 2a.2  TOPICS ──────────────────────────────────────────────────────────────
\subsubsection{Topics}
We use the 65 TREC BioGen 2024 topics (IDs~116--180) distributed in
\texttt{BioGen2024topics.json}.  Each topic contains three fields:
a short keyword \texttt{topic} (2--6 words), a natural-language clinical
\texttt{question} (10--20 words), and a longer \texttt{narrative} providing
retrieval context (30--70 words).  As shown in the evaluation
(§\ref{sec:eval_results}), the combination of \texttt{topic+question}
yields the best retrieval performance; the \texttt{narrative} tends to
introduce instructional phrasing that adds noise.

% ── 2a.3  GROUND TRUTH ─────────────────────────────────────────────────────────
\subsubsection{Ground Truth}
Relevance judgements are derived from the file
\texttt{biogen\_2024\_\allowbreak submissions\allowbreak .json},
which contains automated
BioGen~2024 system submissions annotated with sentence-level citation
assessments (the \texttt{evidence\_relation} field).
We construct two relevance scales — \emph{binary}
(supporting\,=\,1, else\,=\,0) and \emph{graded}
(supporting\,=\,5, neutral\,=\,2, else\,=\,0) — to support
different evaluation metrics.  Full construction details and the
train/test split protocol appear in §\ref{sec:eval_setup}.

% ── 2a.5  INDEX DESIGN ────────────────────────────────────────────────────────
\subsubsection{Index Design}

All documents are stored in a single OpenSearch index
(\texttt{usernlp03}), hosted on the shared course instance at
\texttt{api.nova\allowbreak search.org:443}~\cite{opensearch},
with 4~shards, 0~replicas,
and refresh disabled during bulk ingestion.
Rather than maintaining one index per retrieval strategy, we create
\emph{one field per parameter configuration} inside this single index.
OpenSearch computes BM25 and language-model scores at index time based on
each field's similarity setting, so every configuration is pre-computed and
available instantly at query time with no on-the-fly recomputation.

The index is built in two passes.  The \textbf{baseline pass} creates the
initial fields used for retrieval demos and field ablation:
\texttt{doc\_id} (keyword, PMID),
\texttt{contents\_bm25\_\allowbreak k12\_b075} (BM25 $k_1$=1.2, $b$=0.75),
\texttt{contents\_lmjm\_07} (LM-JM $\lambda$=0.7),
\texttt{contents\_lmdir\_2000} (LM-Dir $\mu$=2{,}000), and
\texttt{embedding\_msmarco} (768-dim HNSW vector).
The \textbf{tuning pass} then extends the live index — via a
close\,/\,add-similarities\,/\,reopen cycle — with the full parameter grid:
24~BM25 fields ($k_1 \times b$), 7~LM-JM fields, 9~LM-Dir fields, and
4~KNN embedding fields (one per encoder), totalling roughly 45~fields.
Field names follow a programmatic convention
(\texttt{contents\_bm25\_k\{k1\}\_\allowbreak b\{b\}},
\texttt{contents\_lmjm\_\allowbreak\{$\lambda$\}}, etc.)\ generated by
\texttt{index\_builder.py}, and the lifecycle method
\texttt{create\_or\_\allowbreak update\_index()} diffs live fields against the
requested set so that additions are idempotent.

We use the \texttt{standard} analyser (tokenise + lowercase) rather than the
\texttt{english} analyser, because Porter stemming tends to mangle biomedical
terms (e.g.\ \emph{hepatitis} $\to$ \emph{hepat}).
For KNN fields, vectors are L2-normalised before indexing, so
\texttt{innerproduct} is equivalent to cosine similarity; HNSW parameters
are \texttt{ef\_construction}$\,=\,$256, $M\!=\!48$,
\texttt{ef\_search}$\,=\,$100.

% ── 2a.6  RETRIEVAL STRATEGIES ────────────────────────────────────────────────
\subsubsection{Retrieval Strategies}

We implement five retrieval strategies covering both sparse (lexical) and
dense (neural) paradigms.  All share a uniform interface —
\texttt{BaseRetriever.search(query, size=100)} — and return the top-100
documents per query as a ranked list of (PMID, score) pairs.

\paragraph{Sparse retrievers:}
\textbf{BM25}~\cite{robertson1994bm25} is a probabilistic term-weighting
model that issues a \texttt{match} query against the appropriate
\texttt{contents\_bm25\_\allowbreak k\{k1\}\_b\{b\}} field.
The default parameters ($k_1$=1.2, $b$=0.75) are tuned via grid search in
§\ref{sec:eval_results}, where $k_1$ controls term-frequency saturation and
$b$ governs length normalisation.
\textbf{LM~Jelinek-Mercer} smooths the document language model via linear
interpolation with the collection model, controlled by a single
parameter~$\lambda$ (default 0.7).
\textbf{LM~Dirichlet}~\cite{lm_retrieval} applies Dirichlet-prior smoothing,
where $\mu$ determines how much probability mass is shifted to the collection
model (default 2{,}000).

\paragraph{Dense retriever:}
\textbf{KNN} performs HNSW approximate nearest-neighbour search on a
768-di\-men\-sion\-al embedding field, using cosine similarity via the
inner product of L2-normalised vectors
(\texttt{ef\_search}\,=\,100).
Four encoder families are compared in §\ref{sec:eval_results}; the
winning model is used for the final \texttt{embedding\_medcpt} field.

\paragraph{Hybrid fusion:}
\textbf{Reciprocal Rank Fusion (RRF)}~\cite{rrf2009} merges two ranked lists
in a score-agnostic manner:
$\text{RRF}(d)=\sum_r \frac{1}{k + \text{rank}_r(d)}$, with $k$\,=\,60
(Cormack default).  We fuse BM25 with KNN\,(MedCPT), chosen because
lexical and semantic signals are largely complementary.

% ── 2a.7  DENSE ENCODER ───────────────────────────────────────────────────────
\subsubsection{Dense Encoder Details}

Our baseline encoder is \texttt{msmarco-\allowbreak distilbert-\allowbreak base-v2}~\cite{reimers2019sbert},
a DistilBERT model fine-tuned on 8.8M MS~MARCO web-passage pairs that
produces 768-dimensional embeddings.  Hidden states are pooled and then
L2-normalised via \texttt{F.normalize(p=2, dim=1)} before indexing.

\paragraph{Pooling-mode comparison:}
Before comparing encoder families, we evaluate three pooling modes —
\emph{mean}, \emph{mean-no-special} (excluding \texttt{[CLS]}/\texttt{[SEP]}),
and \emph{CLS} (first token only) — for every encoder using brute-force exact
cosine in Python (no HNSW approximation) to isolate model quality from
index-level noise.  Table~\ref{tab:pooling} summarises the results: CLS
pooling works best for MedCPT and SPECTER-2, while mean-no-special is
preferable for msmarco and multi-qa.  Each encoder's best pooling mode is
locked for all subsequent experiments.

\paragraph{Domain-matched encoder:}
The primary dense model is
\texttt{ncbi/\allowbreak MedCPT-\allowbreak Query-\allowbreak Encoder}
+
\texttt{ncbi/\allowbreak MedCPT-\allowbreak Article-\allowbreak Encoder}~\cite{medcpt2023},
an \emph{asymmetric} dual-encoder trained on 23M PubMed search
click-through pairs.  The query encoder (max-length 64, CLS pooling) handles
queries, while the article encoder (max-length 512, CLS pooling) processes
documents — an exact domain match for our PubMed retrieval task.
We also evaluate \texttt{multi-qa-mpnet-\allowbreak base-cos-v1}~\cite{reimers2019sbert}
and \texttt{allenai/\allowbreak specter2}; full comparison results appear in
§\ref{sec:eval_results}.
